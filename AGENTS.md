# Правила работы агента в репозитории MetaEngine

Этот репозиторий ведется по согласовательному процессу. Агент не должен сразу
переходить к кодингу, если задача требует продуктового или архитектурного
решения.

## Быстрый контекст для нового чата

Перед началом работы агент должен прочитать:

1. `AGENTS.md` — обязательные правила работы.
2. `README.md` — пользовательский обзор проекта и запуск.
3. `docs/PROJECT_CONTEXT.md` — полный контекст проекта и продуктовые решения.
4. Тематические документы из `docs/`, если задача относится к конкретному модулю:
   - `docs/STRATEGIES.md` — торговые стратегии и модульная структура;
   - `docs/CSV_EXPORT.md` — экспорт CSV.

`AGENTS.md` — не единственный большой источник контекста. Это быстрый вход и
обязательные правила. Подробный контекст хранится в `docs/PROJECT_CONTEXT.md`,
а детали модулей — в тематических документах `docs/`.

## Обязательный порядок работы

1. Сначала обсудить задачу с пользователем: цель, входные данные, ожидаемый
   результат, ограничения, риски и критерии приемки.
2. Сформулировать условие задачи или краткий план реализации и явно спросить:
   "Устраивает ли такое условие?"
3. Начинать кодить только после подтверждения пользователя.
4. После реализации показать результат: что изменено, как проверено, какие
   тесты пройдены или почему тесты не запускались.
5. После проверки предложить запушить результат в GitHub. Push выполняется
   только после согласия пользователя, если оно не было заранее дано в текущей
   задаче.
6. Деплой выполнять отдельным шагом после push и отдельного подтверждения
   пользователя.

## Документация после изменений

После каждого функционального изменения агент должен обновить релевантные
документы:

- `README.md`, если изменился пользовательский сценарий, запуск или структура проекта;
- `AGENTS.md`, если изменились правила работы агента, workflow или быстрый контекст;
- `docs/PROJECT_CONTEXT.md`, если изменились общий контекст, архитектура или продуктовые решения;
- тематические документы в `docs/`, если изменился конкретный модуль.

Новые крупные темы лучше выносить в отдельные документы в `docs/`, чтобы
`docs/PROJECT_CONTEXT.md` не превращался в один огромный файл.

## PR workflow для solo-разработки

- В рамках одного связанного этапа работ предпочтительно вести один активный PR
  и добавлять изменения новыми осмысленными коммитами.
- Новый PR открывать только для нового самостоятельного этапа, крупного
  эксперимента или после merge предыдущего PR в `main`.
- Не мержить PR в `main` без явного подтверждения пользователя.
- Перед merge проверить `git status`, тесты и состав PR.
- Агент должен советовать пользователю, когда лучше продолжить текущий PR,
  когда зацементировать изменения merge в `main`, а когда открыть новый PR.

## Git и безопасность

- Не откатывать чужие изменения в git без прямого запроса пользователя.
- Не использовать destructive-команды вроде `git reset --hard` без явного
  разрешения.
- Коммиты должны быть осмысленными и небольшими по смысловому объему.
- Перед push нужно проверить `git status` и убедиться, что в коммит попали
  только относящиеся к задаче изменения.

## Проектные defaults

- Основной backend-стек: ASP.NET Core / C# на .NET 10 LTS.
- Основная база данных: PostgreSQL.
- Расчеты запускаются асинхронно и сохраняют результат в базе.
- API должен отдавать фронтенду JSON-контракты, включая Plotly-compatible
  `data/layout/config`.
- Python допустим как reference/prototyping инструмент для проверки формул, но
  не как production runtime MVP.
