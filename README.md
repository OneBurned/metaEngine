# MetaEngine

MetaEngine сейчас находится на этапе **локальной лаборатории для проверки стратегий, пресетов и базовых расчетов**.

Цель текущей версии — быстро проверять CSV-файлы, собирать из них пресеты с весами и периодами, считать базовые ряды и смотреть результат в браузере.

Основные документы проекта:


> Важно: это пока **файловый прототип** на Node.js для проверки логики. По ТЗ production-версия позже должна быть на ASP.NET Core / C# / PostgreSQL.

- Хранение пресетов в `samples/presets`.
- Удаление пресетов через интерфейс и API.
- Расчет одной стратегии или пресета за выбранный период.
- Расчет базовых рядов:
  - `diff`;

## Формат CSV стратегии

Ожидается CSV без заголовков:

```csv
1734606000000,0
1734609600000,0.0001422356863014
1734613200000,-0.0005010288408471
```

То есть:

```text
timestamp,value
```

Где:

- первая колонка — timestamp;
- вторая колонка — `diff` или `accum`;
- тип второй колонки выбирается в интерфейсе при загрузке.

После загрузки стратегия сохраняется в `samples/strategies` уже в нормализованном формате:

```csv
timestamp,diff
1734606000000,0
1734609600000,0.0001422356863014
```

Если был загружен `accum`, он пересчитывается в `diff`, и дальше система работает уже через `diff`.

## Время и даты

В интерфейсе даты показываются в формате:

```text
YYYY-MM-DD HH:MM
```

Примеры:

```text
2026-04-30 14:00
2025-04-09 23:00
2021-10-03 23:00
```

Timestamp читается как Unix timestamp в миллисекундах. Внутри используется UTC-форматирование без дополнительного пользовательского сдвига часового пояса.

## Правило отсутствующих данных

Единое правило:

```text
если данных нет для diff → берем 0
если данных нет для accum → берем предыдущее известное значение
```

На практике после загрузки все стратегии хранятся как `diff`, поэтому при расчетах отсутствующие значения дают `diff = 0`.

Пример для `diff`:

```csv
timestamp,diff
2024-01-06 00:00,0
2024-01-06 01:00,-0.0003
2024-01-06 03:00,0.0002
```

Для расчета это будет обработано так:

```csv
timestamp,diff
2024-01-06 00:00,0
2024-01-06 01:00,-0.0003
2024-01-06 02:00,0
2024-01-06 03:00,0.0002
```

Пример для `accum`:

```csv
timestamp,accum
2024-01-06 00:00,0.1
2024-01-06 01:00,0.2
2024-01-06 03:00,0.3
```

Логически пропуск заполняется так:

```csv
timestamp,accum
2024-01-06 00:00,0.1
2024-01-06 01:00,0.2
2024-01-06 02:00,0.2
2024-01-06 03:00,0.3
```

Потом `accum` пересчитывается в `diff`.

## Стратегии

Стратегия — это CSV-файл, сохраненный в:

```text
samples/strategies
```

Через интерфейс можно:

- загрузить стратегию;
- выбрать, во второй колонке `diff` или `accum`;
- посмотреть период данных, шаг и количество пропусков;
- удалить стратегию.

Если стратегия используется в пресете и ее удалить:

- пресет не удаляется;
- при расчете по этой стратегии будут нули;
- если позже загрузить стратегию с тем же именем, она снова начнет участвовать в расчете пресета.

## Пресеты

Пресет — это рецепт из стратегий, весов и периодов.

Пример:

```text
preset_1
- strategy_a: вес 100%, период 2024-01-01 00:00 — 2025-01-01 00:00
- strategy_a: вес 10%, период 2025-01-01 00:00 — до конца
- strategy_b: вес 25%, период 2024-06-01 00:00 — до конца
```

Веса пользователь вводит в процентах:

```text
100 = 100%
25 = 25%
-20 = -20%
150 = 150%
```

Внутри расчета веса применяются как доли:

```text
100% → 1.0
25% → 0.25
-20% → -0.2
150% → 1.5
```

Пресеты хранятся в:

```text
samples/presets
```

Через интерфейс можно:

- создать пресет;
- добавить строки пресета;
- использовать одну стратегию несколько раз в разные периоды;
- поставить «До конца» вместо даты окончания;
- сохранить пресет;
- перезаписать существующий пресет после подтверждения;
- удалить пресет.

При удалении пресета стратегии не удаляются.

## Ребалансировки

Ребалансировка — это когда одна стратегия участвует в одном пресете несколько раз, но с разными периодами и весами.

Так можно:

```text
strategy_a: 100%, 2024-01-01 00:00 — 2025-01-01 00:00
strategy_a: 10%,  2025-01-01 00:00 — до конца
```

Так нельзя:

```text
strategy_a: 100%, 2024-01-01 00:00 — 2025-01-01 00:00
strategy_a: 10%,  2024-06-01 00:00 — 2025-06-01 00:00
```

Потому что периоды одной и той же стратегии пересекаются, и непонятно, какой вес брать.

## Расчет

В блоке «Расчет» можно выбрать:

- стратегию;
- пресет.

Потом нужно указать период:

```text
Период с
Период по
```

Расчет всегда строится через `diff`.

Для пресета формула такая:

```text
preset_diff = сумма(strategy_diff × weight)
```

После этого из общего `diff` строятся:

- `accum`;
- `hwm`;
- `dd`;
- `mdd`.

## Метрики и ряды

### diff

Доходность за конкретный шаг.

### accum

Накопленная доходность за выбранный период.

На первой точке выбранного периода `accum = 0`.

### hwm

High Water Mark — максимум накопленной доходности на текущий момент.

### dd

Drawdown — текущая просадка в каждой точке.

### mdd

Max Drawdown series — худшая просадка, накопленная к каждой точке времени.

Пример:

```text
DD:
0
-0.02
-0.05
-0.01
-0.08
-0.03

MDD:
0
-0.02
-0.05
-0.05
-0.08
-0.08
```

То есть `DD` может восстанавливаться, а `MDD` показывает худшую уже случившуюся просадку.

## График

На графике есть тумблеры:

- `diff`;
- `accum`;
- `hwm`;
- `dd`;
- `mdd`.

Цвет рядом с тумблером совпадает с цветом линии на графике.

## API

### Получить стратегии

```http
GET /api/strategies
```

### Загрузить стратегию

```http
POST /api/strategies
```

Multipart form-data:

- `file` — CSV-файл;
- `name` — имя стратегии;
- `valueType` — `diff` или `accum`.

### Удалить стратегию

```http
DELETE /api/strategies/{file}
```

### Получить пресеты

```http
GET /api/presets
```

### Создать или перезаписать пресет

```http
POST /api/presets
```

JSON body:

```json
{
  "name": "preset_1",
  "overwrite": false,
  "items": [
    {
      "strategy": "strategy_a.csv",
      "weightPercent": 100,
      "date_from": "2024-01-01 00:00",
      "date_to": "2025-01-01 00:00",
      "untilEnd": false
    }
  ]
}
```

Если `untilEnd = true`, то `date_to` сохраняется как `null`.

### Удалить пресет

```http
DELETE /api/presets/{name}
```

### Запустить расчет

```http
POST /api/calculate
```

JSON body:

```json
{
  "targetType": "preset",
  "targetName": "preset_1",
  "periodFrom": "2024-01-01 00:00",
  "periodTo": "2025-01-01 00:00"
}
```

Для расчета одной стратегии:

```json
{
  "targetType": "strategy",
  "targetName": "strategy_a.csv",
  "periodFrom": "2024-01-01 00:00",
  "periodTo": "2025-01-01 00:00"
}
```

## Важные ограничения текущей версии

- Файлы стратегий ожидаются как две колонки: `timestamp,value`.
- Текущая загрузка принимает числовые значения как decimal-доходность. Значение `0.015` означает `1.5%`.
- Если загрузить `1.5` без знака `%`, это будет прочитано как `150%`, а не как `1.5%`.
- В дальнейшем нужно добавить явный выбор формата значений: decimal или percent.

## Обновление терминов: портфолио и торговые стратегии

С этого этапа CSV-файлы доходности называются **портфолио**, а слово **стратегии** используется для торговых правил поверх уже рассчитанного портфолио или пресета.

Текущая структура хранения:

```text
samples/portfolios   CSV портфолио в нормализованном формате timestamp,diff
samples/presets      JSON-пресеты из портфолио, весов и периодов
samples/strategies   JSON-конфиги торговых стратегий
samples/runs         JSON-результаты расчетов
```

Код торговых стратегий лежит отдельно от пользовательских JSON-конфигов:

```text
strategies/          модули торговых стратегий: RSI и MDD Mean Reversion
samples/strategies   сохраненные JSON-конфиги пользователя
```

Подробнее: `docs/STRATEGIES.md`.

В блоке “3. Расчет” пользователь выбирает портфолио или пресет и период. Блок “Стратегии” включается отдельным тумблером и применяет торговую стратегию к уже выбранному расчету.

Кнопка расчета стратегии активна только после актуального расчета блока “3. Расчет”. Если расчета нет или параметры блока 3 изменились, кнопка disabled и показывает подсказку, почему стратегию пока нельзя запустить. Во время расчета текст кнопки меняется на “Рассчитывается...” с анимированными точками.

Если в блоке “3. Расчет” меняются даты и панель стратегий открыта, даты периода стратегии автоматически становятся такими же.

Первая торговая стратегия — RSI:

- RSI считается по equity-кривой `1 + accum`;
- на графике доходность отображается как `accum`, то есть стартовая точка равна `0%`;
- параметры по умолчанию: период `14`, buy `30`, sell `70`;
- верхний/нижний уровень графика берутся из `Продать на` / `Купить на`; отдельной baseline-настройки больше нет;
- правило торговли long-only: покупка происходит при пересечении `buyLevel` сверху вниз (`previous RSI > buyLevel && current RSI <= buyLevel`), продажа — при пересечении `sellLevel` снизу вверх (`previous RSI < sellLevel && current RSI >= sellLevel`);
- первая точка периода стратегии не является сигнальной, а повторные покупки/продажи игнорируются, если позиция уже в нужном состоянии;
- сигнал исполняется со следующей точки, чтобы не было подглядывания в будущее;
- если период стратегии выходит за период расчета, показывается предупреждение, а отсутствующие данные заполняются по общему правилу.

Вторая торговая стратегия — MDD Mean Reversion:

- сигнал строится по текущей просадке `dd` исходного ряда;
- количество точек входа настраивается в поле `Кол-во входов`;
- у каждого входа есть свой уровень DD и вес в процентах;
- поле `Макс. общий вес, %` ограничивает суммарный вес всех входов, например `100` означает не больше полной позиции;
- один общий уровень выхода закрывает все набранные входы при восстановлении DD;
- входы и выход исполняются со следующей точки, без подглядывания в будущее.

После расчета стратегии UI показывает:

1. исходный график портфолио/пресета;
2. RSI-подграфик с линиями уровней;
3. отдельный итоговый график и таблицу доходности торговой стратегии.

В блоке “Итог торговли по стратегии” исходный ряд и результат стратегии сведены на один сравнительный Plotly-график. У исходного ряда и результата есть свои галочки `diff/accum/hwm/dd/mdd`. Общий выбор высоты растягивает сравнительный график и RSI вместе, чтобы было проще рассмотреть детали. На графиках можно выделять мышкой участок по X для zoom, двигать масштаб через Plotly pan/zoom и сбрасывать масштаб кнопкой **“Сбросить масштаб”** или двойным кликом. Сравнительный график и RSI синхронизируют X-диапазон.

## Оптимизация стратегий

В блоке “Стратегии” есть оптимизатор RSI и MDD Mean Reversion. Он использует те же правила входа/выхода, что и обычный расчет стратегии, и запускает много прогонов с разными параметрами.

Для RSI перебираются:

- `RSI период`;
- `Купить на`;
- `Продать на`.

`Верхний уровень`, `Нижний уровень` и `Базовая линия` больше не являются отдельными RSI-параметрами: верхний/нижний уровень берутся из `Продать на` / `Купить на`.

Для MDD перебираются:

- `Вход 1..N`;
- `Вес 1..N`;
- `Выход`.

`N` берется из поля `Кол-во входов`, сейчас максимум 10. Оптимизатор отбрасывает MDD-кандидаты, у которых сумма весов входов больше `Макс. общий вес, %`.

У MDD-оптимизатора есть два режима параметров:

- `Простой` — пользователь задает общий диапазон входов, минимальную дельту между соседними входами, общий диапазон веса на вход и максимальный общий вес;
- `Детальный` — пользователь задает отдельный диапазон DD и веса для каждого входа.

В обоих режимах оптимизатор нормализует кандидаты к одному формату `entry1..entryN`, `weight1..weightN`, `exitLevel`. В простом режиме входы сортируются и проходят проверку минимальной дельты между соседними уровнями.

Для MDD по умолчанию используется режим `Случайный`: оптимизатор берет не всю гигантскую сетку, а ограниченное число кандидатов из поля `Кандидатов` и фиксированный `Seed` для воспроизводимости. Это позволяет задавать широкие диапазоны вроде `0..80` по входам и перебирать веса без падения по памяти. `Полный перебор` оставлен для маленьких диапазонов.

Оптимизатор показывает таблицу лучших прогонов:

- параметры;
- score;
- итоговый `accum`;
- `MDD`;
- количество покупок и продаж.

Заголовки числовых столбцов в таблице кликабельны. Первый клик сортирует по убыванию, повторный клик по тому же столбцу — по возрастанию.

Во время оптимизации интерфейс показывает процесс пересчета:

- сколько прогонов уже выполнено из общего числа;
- сколько комбинаций параметров уже обработано;
- сколько семплов используется;
- текущий набор параметров;
- лучший score на текущий момент.

Кнопка **“Остановить оптимизацию”** мягко завершает текущий прогон, больше не берет новые комбинации и показывает таблицу результатов по уже посчитанным прогонам.

Жесткого ограничения на количество прогонов нет. Если сетка получилась большой и расчет идет долго, используй кнопку остановки.

Для длинных оптимизаций backend держит в памяти только лучшие `Показать лучших` результатов. Счетчики прогресса продолжают считать всю сетку, но неудачные прогоны не копятся в объекте задачи.

Для отбора можно задать фильтры:

- `Макс. просадка, %` — например `20` означает, что строки с худшей MDD ниже `-20%` не попадут в результаты;
- `Мин. трейдов` — минимальная сумма покупок и продаж по всем семплам;
- `Мин. прибыльных семплов` — сколько семплов должны закрыться с положительным accum.

Фильтры применяются на backend во время оптимизации: прогон считается и учитывается в progress, но если он не прошел отсечение, он не попадает в top-results.

Оптимизатор можно запускать по семплам. Если указать `Семплов = 3`, выбранный период делится на 3 последовательных куска по точкам исходного ряда. Каждая комбинация параметров считается на каждом семпле отдельно. В таблице показываются результаты по каждому семплу и общая метрика устойчивости:

Перед перебором backend готовит семплы, а RSI считает и кэширует отдельно для каждой пары `семпл + RSI период`. После этого комбинации `Купить на` / `Продать на` считаются по готовому RSI-ряду без пересборки полного графика на каждый прогон.

```text
устойчивость = худший score среди семплов
```

Так выше ранжируются настройки, которые работают стабильнее на разных кусках истории, а не только на одном полном треке.

В таблице также есть `Сцепл. accum`: это доходность семплов, перемноженная как последовательные периоды:

```text
(1 + accum sample 1) * (1 + accum sample 2) * ... - 1
```

Score сейчас считается как Recovery:

```text
score = итоговый accum / abs(MDD)
```

Если `MDD = 0`, деление на ноль не выполняется, score обрабатывается отдельно.
## Экспорт CSV

В local lab есть отдельный блок **“Экспорт CSV”**. Кнопка открывает popup-окно в стиле кастомного date-picker.

В popup выбирается источник экспорта:

- сохраненное портфолио;
- текущий исходный результат расчета;
- текущий результат торговой стратегии.

Колонка `timestamp` / дата всегда включена и не отключается. Остальные колонки выбираются независимыми тумблерами:

- `diff`;
- `accum`;
- `hwm`;
- `dd`;
- `mdd`.

Можно экспортировать любые комбинации, например `timestamp,mdd`, `timestamp,accum` или `timestamp,diff,accum,hwm,dd,mdd`. Для текущего исходного результата и текущего результата торговой стратегии CSV собирается в браузере из уже рассчитанных строк.

Для сохраненного портфолио frontend вызывает backend endpoint с параметром `columns`, например:

```text
GET /api/portfolios/portfolio_a.csv/export?columns=timestamp,mdd
```

Сохраненные портфолио лежат как `timestamp,diff`, поэтому сервер при экспорте пересчитывает полный ряд портфолио и отдает только выбранные колонки: `accum`, `hwm`, `dd`, `mdd` строятся из сохраненного `diff`.

Имена скачиваемых файлов отражают источник и выбранные колонки, например:

```text
portfolio_timestamp_mdd.csv
base_result_timestamp_accum.csv
strategy_result_timestamp_diff_accum_mdd.csv
```
