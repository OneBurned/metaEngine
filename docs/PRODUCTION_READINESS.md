# MetaEngine production readiness

Этот документ описывает путь от текущей Node.js local lab к production-версии
MetaEngine. Он фиксирует целевую архитектуру, требования к безопасности,
хранению данных, расчетам, оптимизатору, тестированию и выпуску.

Документ является согласованным планом подготовки. Он не означает, что текущий
`main` уже можно публиковать или что production-деплой разрешен без отдельной
проверки и подтверждения пользователя.

## 1. Граница продукта

Production MVP должен позволять пользователю:

- загружать ряды доходностей портфолио;
- создавать пресеты с весами и периодами;
- рассчитывать `diff`, `accum`, `hwm`, `dd`, `mdd`;
- применять RSI, MDD Mean Reversion и MDDGrid;
- запускать оптимизацию на нескольких последовательных семплах;
- останавливать оптимизацию после завершения текущего кандидата;
- сохранять выбранный результат оптимизации как версионируемую мета-стратегию;
- повторно использовать сохраненную мета-стратегию как источник доходностей при
  создании нового пресета;
- сравнивать исходный ряд и результат стратегии;
- сохранять, сортировать, фильтровать и повторно применять результаты;
- экспортировать расчетные данные в CSV.

Сохраненная мета-стратегия является полноценным переиспользуемым расчетным
источником. Вместе с ней сохраняются исходное портфолио или пресет, точная
версия параметров из оптимизатора, период, таймфрейм, версия calculation engine
и ссылка на результат оптимизации. В строке нового пресета пользователь может
выбрать обычное портфолио или сохраненную мета-стратегию и задать ей вес и период
участия.

Production MVP не размещает реальные биржевые заявки и не управляет средствами.
Это аналитический сервис и бэктестер мета-стратегий. Интеграция с торговым
исполнением является отдельным будущим этапом.

### Реализовано в P2c

Production foundation уже позволяет авторизованному пользователю сохранить и
прочитать неизменяемую версию пресета, состоящую из точных версий портфелей.
Периоды одной версии источника не пересекаются, вес задается decimal и может
создавать плечо без общего лимита. Доменное ядро умеет построить результирующий
`diff/accum/hwm/dd/mdd` ряд пресета и проверяется shared golden fixture.
P3/P4 now cover the public calculation workflow, artifact storage and the first
production UI. P5 adds manual strategy calculations; P6 adds RSI and MDD
optimizer workflows. P8 adds lease-based recovery and retry controls; Worker
replica deployment and capacity settings are now available through the Compose
foundation. TLS, backups, monitoring and release operations remain later work.

### Реализовано в P5a/P5b

RSI, MDD Mean Reversion и MDDGrid выполняются отдельными Worker-запусками и
сохраняются как неизменяемые strategy results. Сохранённый результат стратегии можно
добавить в новый пресет рядом с портфелями, задать ему вес и период, а затем
поставить этот пресет в очередь через production UI. Production UI для
RSI- и MDD-оптимизаций уже доступен: пользователь задает ranges, samples и
filters, видит progress, останавливает job и применяет candidate как обычную
стратегию. Failed и interrupted задачи можно повторить. Production CSV export
и публичные deployment-операции остаются следующими этапами.

### Реализовано в P3

Пользователь может поставить базовый расчет точной версии портфеля или пресета
в очередь. Отдельный Worker обрабатывает его вне HTTP-запроса, сохраняет
итоговые метрики и неизменяемый artifact `timestamp,diff`; API отдает статусы,
warnings и страницы результата. P4 now adds the UI for portfolio import and
base calculation. Subsequent P5/P6/P8 stages add production strategies,
optimization, manual retry and restart recovery; these capabilities are now in
`main`.

## 2. Уровни окружений

| Окружение | Назначение | Допустимое использование |
| --- | --- | --- |
| Local lab | Быстрая проверка формул и UI | Только локальная разработка |
| Staging | Проверка production-сборки и миграций | Закрытый доступ, тестовые или копии данных |
| Production | Постоянная работа пользователей | Только после прохождения всех release gates |

Текущий Node.js сервер может оставаться reference-реализацией формул и локальной
лабораторией. Он не должен становиться публичным production backend.

## 3. Целевая архитектура

Основной стек уже зафиксирован правилами проекта:

- ASP.NET Core / C# на .NET 10 LTS;
- PostgreSQL;
- асинхронные расчеты и оптимизации;
- JSON API для frontend;
- Plotly-compatible `data/layout/config` для графиков.

Рекомендуемая структура solution:

```text
MetaEngine.Api           HTTP API, авторизация, валидация, выдача результатов
MetaEngine.Application   сценарии расчетов и управление заданиями
MetaEngine.Domain        формулы, стратегии, метрики и доменные контракты
MetaEngine.Infrastructure PostgreSQL, очередь, файлы, логи и внешние сервисы
MetaEngine.Worker        расчеты и оптимизации вне HTTP-процесса
MetaEngine.Strategies.Abstractions общий контракт модулей мета-стратегий
MetaEngine.Strategies.Rsi           модуль RSI
MetaEngine.Strategies.MddMeanReversion модуль MDD Mean Reversion
MetaEngine.Strategies.MddGrid       модуль MDDGrid
MetaEngine.Tests         unit, golden, integration и API tests
```

Поток выполнения:

```text
Browser -> HTTPS/reverse proxy -> MetaEngine.Api -> PostgreSQL
                                      |
                                      v
                                  job queue
                                      |
                                      v
                              MetaEngine.Worker
                                      |
                                      v
                         PostgreSQL / result storage
```

API и worker должны запускаться как отдельные процессы. Долгий расчет не должен
занимать HTTP request и не должен блокировать работу интерфейса.

### Модульный контракт мета-стратегий

RSI, MDD Mean Reversion и MDDGrid являются реализациями общего контракта, а не
особыми режимами ядра. Новые стратегии должны подключаться отдельными модулями
без изменения общего API, очереди заданий, хранения результатов и механизма
семплирования.

Каждый модуль мета-стратегии описывает:

- стабильный `strategyType`, отображаемое имя и `schemaVersion`;
- схему параметров расчета и значения по умолчанию;
- схему диапазонов и ограничений для оптимизатора;
- проверку и нормализацию параметров;
- подготовку данных, которые можно рассчитать один раз до перебора;
- расчет одного набора параметров с поддержкой отмены;
- выходной ряд `timestamp,diff`;
- дополнительные индикаторы, сигналы, состояния и колонки таблицы;
- описание Plotly traces и вспомогательных графиков;
- миграцию сохраненных параметров со старой версии схемы.

Общее ядро отвечает за:

- получение и версионирование исходного ряда;
- разбиение на семплы;
- очередь, прогресс, stop, retry и checkpoint;
- потоковый перебор кандидатов;
- общие метрики, фильтры, ранжирование и top-N;
- сохранение выбранного результата и аудит.

Модуль отвечает только за специфичную для стратегии подготовку, генерацию
допустимых параметров и торговую симуляцию. В API и worker не должно появляться
ветвление вида `если RSI`, `если MDD` для запуска общего сценария.

Модули регистрируются через dependency injection при сборке приложения. Для
production MVP не используется загрузка неизвестных DLL во время работы. Это
не мешает добавлять новые встроенные модули отдельными проектами solution.

### Сохранение результата мета-стратегии

После того как пользователь выбрал результат оптимизации и сохранил
мета-стратегию, система выполняет полный расчет выбранных параметров на
исходном периоде и создает новую неизменяемую версию результата.

Канонический сохраняемый ряд содержит только:

```text
timestamp,diff
```

Из `diff` детерминированно строятся `accum`, `hwm`, `dd` и `mdd`. Эти производные
ряды можно кэшировать для быстрого отображения и экспорта, но кэш не является
источником истины: его можно удалить и пересчитать из сохраненного `diff`.

Вместе с рядом сохраняются:

- `finalAccum`, итоговый HWM и MDD;
- количество точек и трейдов;
- исходное портфолио или пресет и его точная версия;
- версия конфигурации мета-стратегии;
- `strategyType`, `schemaVersion` и нормализованный JSON параметров;
- период и таймфрейм расчета;
- версия calculation engine;
- checksum ряда `timestamp,diff`;
- идентификатор optimization job и выбранного результата, если параметры были
  получены оптимизатором.

Оптимизатор не сохраняет полные ряды всех кандидатов. Во время поиска он хранит
только агрегированные метрики и top-N результатов. Полный `timestamp,diff`
строится после явного выбора и сохранения результата пользователем.

Сохраненная мета-стратегия содержит две связанные части:

1. Версионируемую конфигурацию, которую можно применить к обновленным исходным
   данным.
2. Неизменяемую версию уже рассчитанного ряда `timestamp,diff`, которую можно
   безопасно использовать как источник в пресете.

Если исходное портфолио или пресет обновились, существующий сохраненный ряд не
меняется автоматически. Пользователь запускает пересчет и получает новую
версию результата. Пресет всегда ссылается на конкретную версию ряда, поэтому
его исторический результат остается воспроизводимым.

Обязательные поля воспроизводимости расчета:

- идентификатор и версия входных данных;
- SHA-256 исходного файла и нормализованного ряда;
- версия calculation engine;
- версия схемы стратегии;
- полный JSON параметров;
- период и таймфрейм;
- правило заполнения пропусков;
- дата запуска, пользователь и workspace;
- seed для случайного поиска;
- идентификатор optimization job и выбранного результата, если стратегия
  сохранена после оптимизации;
- статус, ошибка и время выполнения.

Зависимости расчетных источников должны быть направленными и не содержать
циклов. Нельзя создать пресет, который прямо или через сохраненную
мета-стратегию зависит сам от себя.

Большие результаты нельзя сохранять одним многомегабайтным JSON-файлом. Summary
и top-N хранятся в PostgreSQL. Длинные расчетные ряды хранятся порционно в БД
или во внешнем объектном хранилище с записью ссылки и checksum в БД.

## 5. Контракты расчетов

До переноса формул в C# нужно зафиксировать golden datasets. На одинаковых
входах Node.js reference и C# engine должны выдавать одинаковые результаты в
пределах согласованной точности.

Формат, числовая точность и текущие машиночитаемые эталоны зафиксированы в
`docs/CALCULATION_CONTRACTS.md` и `test/fixtures/golden`.

Golden tests должны покрывать:

- `diff -> accum -> hwm -> dd -> mdd`;
- `accum -> diff`, включая первую точку и пропуски;
- пресеты, веса, открытые периоды и ребалансировки;
- границы `1m`, `5m`, `15m`, `1h`, `1d`, `1M`, `1Y`;
- RSI, исполнение сигнала со следующей точки и отсутствие lookahead;
- MDD Mean Reversion, локальный MDD, целевые веса и TP;
- MDDGrid, добавочные лоты, индивидуальные TP и повторный вход по новому DD-cross;
- разделение на семплы;
- Recovery score, сцепленную доходность и фильтры оптимизатора;
- остановку после текущего кандидата;
- экстремальные значения, пустые периоды и пропущенные данные.

Перед production нужно отдельно подтвердить определения:

- что считается одним трейдом;
- входят ли комиссии, slippage и стоимость плеча;
- допустимы ли доходности меньше или равные `-100%`;
- как интерпретировать параметры в процентах и decimal;
- как ранжировать нулевую просадку и нулевое число сделок;
- является ли последний незакрытый вход завершенным трейдом.

## 6. Безопасный оптимизатор

Главное правило: количество прогонов не ограничивается произвольным небольшим
числом, но один запуск не может получить неограниченные ресурсы всего сервера.

Оптимизатор является общим orchestration engine. Он не знает параметры RSI,
уровни MDD или будущие параметры других стратегий. Зарегистрированный модуль
передает оптимизатору валидатор пространства поиска, поток кандидатов,
подготовленные данные семплов и функцию расчета одного кандидата.

Требуемая модель:

1. API проверяет диапазоны и оценивает количество комбинаций.
2. API создает `optimization_job` и возвращает `202 Accepted` с `jobId`.
3. Worker берет задание из очереди.
4. Кандидаты генерируются потоком, а не создаются целиком в памяти.
5. В памяти хранится только текущий кандидат, подготовленные семплы и top-N.
6. Прогресс периодически записывается в БД.
7. При запросе остановки текущий кандидат завершается, новые не берутся.
8. После перезапуска worker задание продолжается с checkpoint или получает
   явно видимый статус `interrupted` с возможностью перезапуска.

Оптимизации производительности:

- базовые ряды семплов рассчитываются один раз до перебора;
- для RSI значения кэшируются по уникальной паре `sample + rsiPeriod`;
- для MDD базовые DD и local MDD рассчитываются один раз на семпл;
- кандидат считает только торговую симуляцию и агрегированные метрики;
- полные строки каждого кандидата не сохраняются;
- top-N поддерживается ограниченной приоритетной очередью;
- график выбранного результата пересчитывается отдельно по запросу пользователя;
- случайный поиск воспроизводится по `seed`;
- параллелизм ограничивается на уровне worker и workspace.

Вместо жесткого лимита прогонов используются:

- очередь заданий;
- лимит одновременно работающих заданий на пользователя/workspace;
- лимит CPU и памяти worker-контейнера;
- настраиваемый timeout или бюджет вычислительного времени;
- предупреждение об оценочном времени до запуска;
- возможность поставить очень большой поиск в долгую очередь;
- административная отмена задания, которое угрожает стабильности сервиса.

Статусы задания:

```text
queued -> running -> completed
                  -> stopping -> stopped
                  -> failed
                  -> interrupted
```

## 7. API

Production API версионируется, например `/api/v1`. Основные группы endpoint:

```text
/api/v1/portfolios
/api/v1/presets
/api/v1/strategy-types
/api/v1/strategies
/api/v1/calculation-runs
/api/v1/optimization-jobs
/api/v1/exports
```

Требования к контрактам:

- OpenAPI является частью сборки;
- `GET /strategy-types` возвращает доступные типы, версии, параметры,
  ограничения оптимизации и описание визуализации;
- входы проходят schema validation;
- ошибки имеют стабильный код, сообщение и `requestId`;
- списки имеют пагинацию;
- большие ряды отдаются страницами или отдельным artifact endpoint;
- длительные операции возвращают job, а не держат HTTP соединение;
- повторная отправка команды поддерживает idempotency key;
- frontend получает Plotly-compatible `data/layout/config`;
- даты передаются в ISO 8601 UTC;
- decimal и percent никогда не определяются по догадке.

## 8. Безопасность

До публичного доступа обязательны:

- аутентификация;
- роли `Admin`, `Researcher`, `Viewer`;
- проверка workspace на каждом чтении и изменении;
- HTTPS на reverse proxy;
- хранение секретов вне репозитория;
- ограничение размера HTTP body, CSV и количества строк;
- rate limiting для API и отдельные квоты оптимизатора;
- защита cookie и CSRF, если используется cookie authentication;
- Content Security Policy и стандартные security headers;
- аудит удаления данных, запуска и остановки расчетов;
- запрет вывода stack trace и внутренних путей пользователю;
- регулярное обновление и сканирование зависимостей и контейнеров.

Новый GitHub-репозиторий должен быть приватным, пока не проведена отдельная
проверка данных. Реальные портфолио, результаты расчетов, дампы БД и секреты не
должны храниться в Git.

## 9. Валидация данных

Загрузка портфолио должна явно принимать:

- `valueType = diff | accum`;
- `valueScale = decimal | percent`;
- таймзону или подтвержденное правило UTC;
- разрешенный набор таймфреймов;
- максимальный размер файла и количество строк из конфигурации окружения.

Сервер отклоняет:

- `NaN`, `Infinity` и пустые числовые поля;
- невалидные или дублирующиеся timestamp;
- даты вне допустимого диапазона;
- несовместимый таймфрейм;
- нечисловые веса и параметры стратегии;
- невалидные MDD уровни, TP и RSI period;
- диапазоны оптимизатора с нулевым или отрицательным шагом.

Предупреждения о пропусках должны сохраняться вместе с запуском и оставаться
доступными после перезагрузки страницы.

## 10. Frontend и графики

Production frontend должен:

- строить формы расчета и оптимизации по descriptor зарегистрированного типа
  стратегии, а не по жестко зашитым формам RSI/MDD;
- использовать Plotly для масштабирования и навигации мышью;
- синхронно масштабировать основной график и RSI/MDD подграфик;
- показывать прогресс, очередь, примерное оставшееся время и остановку задания;
- восстанавливать состояние задания после обновления страницы;
- загружать таблицы страницами, а не строить весь HTML сразу;
- использовать серверную агрегацию или downsampling с сохранением экстремумов;
- корректно работать без горизонтального переполнения на ширине от `360px`;
- показывать понятные ошибки без технического stack trace;
- не хранить критичное состояние только в памяти вкладки.

Для длинного ряда браузер не должен получать и отрисовывать все точки, если
пользователь смотрит общий масштаб. При увеличении выбранного диапазона frontend
запрашивает более подробные данные только для этого диапазона.

## 11. Эксплуатация

Production поставка включает:

- Dockerfile для API и worker;
- migrations PostgreSQL;
- конфигурацию только через environment/secrets;
- `/health/live` и `/health/ready`;
- graceful shutdown API и worker;
- структурированные JSON logs с `requestId`, `userId`, `workspaceId`, `jobId`;
- метрики очереди, длительности, ошибок, CPU, памяти и размера БД;
- алерты на ошибки, нехватку места и зависшие задания;
- автоматический backup PostgreSQL;
- проверенный сценарий restore;
- политику retention для рядов и результатов оптимизации;
- rollback на предыдущий образ приложения;
- отдельные staging и production настройки.

Минимальные эксплуатационные решения, которые нужно согласовать до деплоя:

| Решение | Что требуется определить |
| --- | --- |
| Hosting | Провайдер, регион, CPU, RAM, disk и сеть |
| Domain | Домен, TLS и доступ к staging |
| Identity | Собственные аккаунты или внешний OIDC provider |
| Backup | RPO, RTO, срок хранения и место копий |
| Retention | Сколько хранить runs, artifacts и audit events |
| Quotas | Параллельные jobs и вычислительный бюджет workspace |
| Data volume | Максимальный CSV, число точек и ожидаемое число пользователей |

## 12. Тестирование

Обязательные уровни:

- unit tests доменных формул;
- golden tests Node.js reference против C# engine;
- integration tests с реальным PostgreSQL;
- API tests авторизации, workspace isolation и валидации;
- worker tests очереди, stop, retry, checkpoint и restart;
- contract tests, обязательные для каждого модуля мета-стратегии;
- тестовый третий модуль, подтверждающий подключение без изменения общего API и
  optimizer engine;
- browser tests основных пользовательских сценариев;
- responsive tests desktop и mobile;
- load tests длинных рядов и нескольких оптимизаций;
- security scan зависимостей, образов и endpoint;
- backup/restore drill на staging.

Критичные формулы должны иметь фиксированную допустимую числовую погрешность.
Изменение формулы или схемы стратегии требует новой версии engine и обновления
golden fixtures.

## 13. Миграция local lab

Порядок переноса:

1. Зафиксировать текущие формулы и golden fixtures.
2. Инвентаризировать `samples/portfolios`, `samples/presets`,
   `samples/strategies`, `samples/runs`.
3. Исключить пользовательские данные из Git и подготовить резервную копию.
4. Реализовать importer с checksum и отчетом по каждой записи.
5. Перенести портфолио, пресеты и стратегии в staging PostgreSQL.
6. Запустить одинаковые расчеты в Node.js и C#.
7. Сравнить summary и ряды до согласованной точности.
8. Провести тестовую оптимизацию RSI и MDD.
9. Проверить stop, restart, backup и restore.
10. После приемки перенести production данные и переключить frontend.

Исторические `samples/runs/*.json` не импортируются автоматически. Сначала нужно
решить, какие результаты действительно ценны. Для выбранных запусков сохраняются
конфигурация, summary и при необходимости полный artifact.

## 14. Этапы реализации

| Этап | Результат |
| --- | --- |
| P0. Контракты | Golden datasets, определения метрик, схема API и модель данных |
| P1. Platform | Solution .NET, PostgreSQL, migrations, auth, workspaces, CI |
| P2. Calculation engine | Импорт, базовые ряды, пресеты, RSI и MDD с parity tests |
| P3. Async jobs | Queue, worker, status and immutable artifacts |
| P4. Production UI | React/TanStack Router, вход, импорт, базовые расчеты и chart brush |
| P5a. Strategy runs | RSI/MDD/MDDGrid runs, saved strategy versions and production UI |
| P5b. Strategy presets | Пресеты из портфелей и сохранённых результатов стратегий |
| P6. Optimizer | Потоковый перебор, кэш семплов, top-N и resource quotas |
| P7. Staging | Миграция данных, load/security/restore tests и приемка |
| P8. Queue reliability | Database lease, retry, recovery and manual restart |
| P9. Deployment foundation | Docker images, controlled migrations and Worker replicas |
| P10. Production | Backup, monitoring, runbook, release и rollback check |

Текущий прогресс:

- P0: созданы общие golden fixtures и Node.js contract tests;
- P1a: создан .NET 10 solution, отдельные API/Worker, общий контракт модулей,
  descriptors RSI/MDD и первые .NET contract tests;
- P1b: добавлены PostgreSQL/EF Core, первая migration, модель users/workspaces,
  версионных портфелей, пресетов и мета-стратегий, jobs и канонических
  `timestamp,diff` artifacts; readiness проверяет БД и migrations;
- P1c: добавлены ASP.NET Core Identity, одноразовый bootstrap владельца,
  HttpOnly cookie + CSRF, роли workspace, изоляция чтения workspace и API tests;
- P1d: PostgreSQL закреплен на major 16, добавлен GitHub Actions CI с migration
  gate, NuGet audit и real-PostgreSQL bootstrap/login/workspace integration test;
- P2a: добавлен workspace-scoped canonical `timestamp,diff` importer,
  неизменяемые версии, raw/series checksum-дедупликация, paginated points и
  PostgreSQL integration coverage;
- P2b: добавлено чистое C#-ядро `diff/accum/hwm/dd/mdd`, source grid с правилом
  `missing diff = 0`, укрупнение до UTC timeframe checkpoints и shared golden
  parity; `diff < -100%` отклоняется, а полная потеря не создает `NaN`;
- P2c/P3/P4: добавлены versioned presets, asynchronous base calculations,
  immutable artifacts, React UI для import/calculation/result/comparison и
  запуск пресетов через очередь;
- P5a/P5b/P5c: добавлены RSI/MDD/MDDGrid strategy runs, versioned saved strategies и
  пресеты, которые используют сохраненную стратегию как источник;
- P6: добавлены streamed RSI/MDD/MDDGrid optimizations с samples, top-N,
  filters, stop, retry и передачей выбранного candidate в strategy run. MDDGrid
  готовит source Accum/HWM/DD один раз на семпл, использует абсолютный DD TP и
  потоково генерирует независимые параметры лотов;
- P8: добавлены атомарные lease-claims PostgreSQL, автоматический retry
  временных ошибок, recovery просроченных Worker-задач, статус `interrupted` и
  ручной повтор расчётов и оптимизаций; несколько Worker-процессов могут
  безопасно читать одну очередь;
- P9: добавлены Compose-образы API/Worker, отдельный migration container,
  две Worker-replica по умолчанию, resource limits и worker ID в JSON-логах;
- еще не реализованы управление участниками, password recovery/2FA/OIDC, rate
  limiting, production CSV export и public-production operations: TLS/reverse
  proxy, backups/restore, metrics/alerts и release runbook.

Каждый этап оформляется отдельным согласованным PR или небольшой серией PR.
Production deploy выполняется отдельным действием после merge и явного
подтверждения пользователя.

## 15. Release gates

Production разрешен только если выполнены все условия:

- [ ] Формулы и определения метрик письменно подтверждены.
- [ ] Golden tests Node.js и C# проходят.
- [ ] Нет данных и секретов в Git или container image.
- [ ] Все endpoint требуют корректные права workspace.
- [ ] API и worker работают раздельно.
- [x] RSI, MDD Mean Reversion и MDDGrid реализуют общий модульный контракт.
- [x] MDDGrid подключена модулем без изменения общего API и очереди; ее
      optimizer остается отдельным следующим этапом.
- [ ] Большой optimizer job не создает всю сетку в памяти.
- [ ] Stop завершает текущего кандидата и сохраняет top-N.
- [ ] Restart API/worker не теряет состояние jobs.
- [ ] Пройдены integration, browser, load и security tests.
- [ ] Мобильная ширина `360px` не имеет общего горизонтального scroll.
- [ ] Настроены health checks, logs, metrics и alerts.
- [ ] Backup создан, restore проверен на staging.
- [ ] Описан rollback и проверен предыдущий container image.
- [ ] Staging приемка подписана пользователем.
- [ ] Production deploy подтвержден пользователем отдельно.
